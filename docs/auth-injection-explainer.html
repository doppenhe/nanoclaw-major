How agents get API access — without ever holding a credential

NanoClaw runs each AI agent in its own Docker container. Agents need to call real APIs (Gmail, GitHub, Stripe, Anthropic…), but an LLM agent is exactly the kind of thing you don't want holding API keys: it can be prompt-injected, it logs its own transcripts, and it executes arbitrary shell commands. The solution: all secrets stay on the host in an encrypted vault, and a man-in-the-middle proxy injects them into outbound HTTPS requests at the network boundary. The container only ever sees a proxy address.

The two worlds

Host machine — trusted

Secrets live here and never leave.

OneCLI Vault (encrypted, at rest)
  • API keys & OAuth access + refresh tokens
  • Host patterns — which credential belongs to which API hostname (e.g. api.stripe.com)
  • Per-agent allow-lists — which agent may use which secret
  • Approval rules for sensitive actions
OneCLI Gateway proxy :10255 · API/UI :10254
  • TLS-intercepting forward proxy with its own private CA
  • Looks up credentials per request, injects auth headers
  • Runs OAuth refresh flows itself — agent never sees tokens
NanoClaw host process
  • Routes chat messages, spawns one container per agent group
  • Registers each agent group with the gateway before spawn
  • Delivers approval requests to a human via DM
container boundary

Agent container — untrusted

Holds zero real credentials. Ever.

Claude agent (Bun runtime, --rm)
  • Makes plain HTTPS calls to the real API URLs — curl https://api.github.com/user/repos
  • No auth headers, no keys, no OAuth dance
  • Taught by a skill file: "just call the API; the gateway handles auth"
Environment (set at docker run)
  • HTTPS_PROXY=http://x:<agent-token>@host.docker.internal:10255
  • The token identifies the agent to the gateway — it is not a service credential and unlocks nothing outside this proxy
  • NODE_EXTRA_CA_CERTS / SSL_CERT_FILE → gateway's CA cert, mounted read-only at /tmp/onecli-gateway-ca.pem
  • Placeholders only: e.g. CLAUDE_CODE_OAUTH_TOKEN=placeholder — the real one is swapped in by the proxy
Credential stubs (for MCP servers)
  • Some tools insist on a credentials file before starting
  • Stub files contain the literal string "onecli-managed" — never a real value
Spawn time → host registers the agent, fetches proxy config from the gateway, passes it as -e env vars + a read-only CA cert mount

Life of a request

What happens when the agent runs curl https://api.stripe.com/v1/charges — with no API key anywhere in the container:

Agent makes a normal HTTPS call container
curl, fetch, git, axios — every standard HTTP client honors HTTPS_PROXY automatically. The agent doesn't do anything special; it just calls the real URL.
Request exits through the gateway proxy gateway :10255
The proxy URL's embedded token tells the gateway which agent is calling. Different agent groups have different tokens — and different secret allow-lists.
TLS is terminated at the proxy gateway
The gateway presents a certificate signed by its own private CA. The container trusts it because the CA cert was mounted in at spawn time. This is a deliberate, local-only man-in-the-middle — it's how the proxy can read and modify the request.
Credential lookup: hostname × agent gateway → vault
The gateway matches api.stripe.com against the vault's host patterns, then checks this agent's allow-list (selective mode = only explicitly assigned secrets; all = any matching secret). No match → the request goes out untouched, or fails with a connect_url the agent can show the user.
Optional: human approval host → your DMs
If a vault rule marks the action as sensitive, the gateway holds the request and emits a pending approval. The NanoClaw host long-polls for these and DMs an approver (scoped admin → global admin → owner). The agent just sees a slow request.
Auth header injected, request re-encrypted, forwarded gateway → real API
The gateway adds Authorization: Bearer sk-live-… (or swaps a placeholder, or runs an OAuth token refresh first) and opens a fresh TLS connection to the real API. The credential exists only on this host-side leg.
Response flows back; the secret never crossed the boundary container
The agent gets a normal API response. Nothing in its environment, filesystem, logs, or chat transcript ever contained a usable credential.

Design details worth stealing

Fails closed
If the gateway is unreachable at spawn time, NanoClaw refuses to start the container at all — there is no "run without credential wiring" fallback that would tempt someone to paste keys into env vars.
Per-agent identity, not per-machine
Every agent group is registered as its own gateway identity with its own token and allow-list. A compromised "fun side-project" agent can't use the secrets assigned to your "production ops" agent.
Revocation is instant and central
The gateway resolves secrets per request, not per session. Rotate or revoke a key in the vault and the very next API call uses (or loses) it — no container restarts, no env-var redeploys.
Prompt injection can't exfiltrate what isn't there
Even if a malicious message tricks the agent into running env or cat ~/.config/* and posting the output, there is nothing to leak — only a proxy address and placeholder strings.
OAuth complexity stays out of the agent
Token refresh, consent flows, and expiry are the gateway's job. When an app isn't connected yet, the error includes a connect_url the agent hands to the human — the agent never participates in the auth flow.
Pragmatic escape hatches exist
Some endpoints don't fit the proxy model (e.g. GitHub's git-over-HTTPS auth). Those are bypassed explicitly with NO_PROXY plus a narrowly scoped token — a visible, deliberate exception rather than a hidden workaround.
