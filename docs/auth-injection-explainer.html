How agents get API access — without ever holding a credential

NanoClaw runs each AI agent in its own Docker container. Agents need to call real APIs (Gmail, GitHub, Stripe, Anthropic…), but an LLM agent is exactly the kind of thing you don't want holding API keys: it can be prompt-injected, it logs its own transcripts, and it executes arbitrary shell commands. The solution: all secrets stay on the host in an encrypted vault, and a man-in-the-middle proxy injects them into outbound HTTPS requests at the network boundary. The container only ever sees a proxy address.

Life of a request

What happens when the agent runs curl https://api.stripe.com/v1/charges — with no API key anywhere in the container. curl, fetch, git, axios all honor HTTPS_PROXY automatically, so the agent does nothing special.

Design details worth stealing

Fails closed
If the gateway is unreachable at spawn time, NanoClaw refuses to start the container at all — there is no "run without credential wiring" fallback that would tempt someone to paste keys into env vars.
Per-agent identity, not per-machine
Every agent group is registered as its own gateway identity with its own token and allow-list. A compromised "fun side-project" agent can't use the secrets assigned to your "production ops" agent.
Revocation is instant and central
The gateway resolves secrets per request, not per session. Rotate or revoke a key in the vault and the very next API call uses (or loses) it — no container restarts, no env-var redeploys.
Prompt injection can't exfiltrate what isn't there
Even if a malicious message tricks the agent into running env or cat ~/.config/* and posting the output, there is nothing to leak — only a proxy address and placeholder strings.
OAuth complexity stays out of the agent
Token refresh, consent flows, and expiry are the gateway's job. When an app isn't connected yet, the error includes a connect_url the agent hands to the human — the agent never participates in the auth flow.
Pragmatic escape hatches exist
Some endpoints don't fit the proxy model (e.g. GitHub's git-over-HTTPS auth). Those are bypassed explicitly with NO_PROXY plus a narrowly scoped token — a visible, deliberate exception rather than a hidden workaround.
